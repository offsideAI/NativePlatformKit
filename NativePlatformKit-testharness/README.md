# NativePlatformKit Test Harness

A **Flutter macOS desktop** app that drives the
[`NativePlatformKit-TestLabs-Android-Playground`](../NativePlatformKit-TestLabs-Android-Playground)
Android app on an emulator and lets a human **step through every screen and capture a screenshot of
each**, saved under [`screenshots/`](screenshots/).

It is a human-in-the-loop tool: the harness boots the emulator, builds & installs the playground,
and presents a guided checklist of all ~80 screens — you navigate the emulator and click **Capture**
for each.

## Requirements

- **macOS** (Apple Silicon) with **Xcode**.
- **Flutter 3.44.2** on `PATH` (`source ~/.zprofile && source ~/.zshrc`).
- **CocoaPods** under **chruby Ruby 3.4.4** (the `Makefile` activates it for builds).
- **Android SDK** at `~/Library/Android/sdk` with the `emulator` binary. Everything else
  (command-line tools, the API 34 system image, the AVD) is installed **from inside the app**.

> The macOS app sandbox is intentionally **disabled** so the harness can launch `adb` / `emulator` /
> `gradlew` and write screenshots to disk. This is a local dev tool, not an App Store build.

## Quick start

```bash
cd NativePlatformKit-testharness
make run          # builds with chruby Ruby active, launches the macOS app
# or: make build  /  make test  /  make analyze  /  make verify
```

Then work through the **guided wizard** in the app:

1. **Environment** — detects the toolchain; installs command-line tools + the API 34 system image if missing.
2. **Emulator** — creates the `npk_pixel_6_api34` AVD and boots it (Pixel 6, API 34, 4 GB RAM, GPU).
3. **Build & Install** — `./gradlew :app:assembleDebug` → `adb install -r` → launches the playground.
4. **Flow Runner** — three-pane guided capture: pick a screen, follow the instructions on the
   emulator, click **Capture** (or **Skip** / add a **Note**). Click **Finish run** to write reports.
5. **Gallery** — browse captured runs; thumbnails by category, click for full size.

## Output layout

```
screenshots/
  runs/<timestamp>/<category>/<screen-id>.png   # per-run archive
  runs/<timestamp>/manifest.json                # device, app (gitSha/versionName), per-screen status
  runs/<timestamp>/report.md, report.csv        # human-readable reports
  latest/<category>/<screen-id>.png             # canonical newest set
```

Resumable run-state lives in `.harness/run-state.json`. Per the project decision, screenshots,
runs, and `.harness/` are **committed** to git.

## Architecture

- **Provider + MVVM**, feature-first (`lib/features/*`), with a native macOS look via `macos_ui`.
- **Native bridge** (`macos/Runner/NpkBridge.swift` + `ProcessRunner.swift`): a `MethodChannel`
  (`npk/commands`) for issuing commands and an `EventChannel` (`npk/events`) that streams live
  stdout/stderr/exit, with per-process cancel/kill. Dart side: `core/native_bridge.dart` +
  `core/command_runner.dart`.
- **Services** wrap external tools through the bridge: `EnvironmentService` (detect + SDK bootstrap),
  `EmulatorService` (AVD lifecycle + boot/readiness), `BuildInstallService` (Gradle + adb),
  `CaptureService` (binary-safe `screencap` → `adb pull`).
- **Flow catalog**: `assets/catalog.json` (generated by `tool/generate_catalog.py`) → loaded and
  validated by `data/flow_catalog/`.
- **Runs**: `RunState`/`RunStore` (resumable `.harness/run-state.json`) + `RunReporter`
  (manifest + Markdown/CSV).

See [`docs/adding-a-screen.md`](docs/adding-a-screen.md) to add or fix a catalog entry, and the
repo-root [`ROADMAP.md`](../ROADMAP.md) for the full implementation breakdown and decisions.

## Make targets

| Target | What |
|---|---|
| `make run` | Build (chruby Ruby active) + launch on macOS |
| `make build` / `make release` | Build the macOS app |
| `make test` | Unit tests |
| `make analyze` | Static analysis |
| `make verify` | analyze + test (CI gate) |
| `make catalog` | Regenerate `assets/catalog.json` |
| `make clean` | `flutter clean` |
